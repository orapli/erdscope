# erdscope engineering quickstart

erdscope is a Python 3.9+ CLI that produces a self-contained interactive ER-diagram HTML file and optional Excel, JSON/config/digest/DBML/Mermaid/PlantUML schema exports. A schema can come from any combination of a live MySQL/PostgreSQL/SQLite database, Rails/Prisma/Django/SQLAlchemy/Laravel model code, typed DBML or Mermaid `erDiagram` input, and declarative JSON/YAML config. The repository’s defining constraint is that development is split into readable source fragments, while distribution remains one zero-required-dependency `erd.py` file.

This wiki documents local HEAD `d028df2` / release `0.9.1` (2026-07-21). The package version is exposed by `erdscope --version` / `-V` and kept aligned with `pyproject.toml` by `tests/test_version.py`.

## Start here

```bash
# zero-setup product smoke test
python3 erd.py demo --no-open
python3 erd.py --version

# use the committed SQLite example
python3 erd.py sqlite:///examples/demo_shop.db -o /tmp/shop.html

# run the normal local suite
python3 -m unittest discover -s tests -v

# verify the committed single-file artifact
python3 tools/build_single_file.py --check
```

Installed usage is equivalent (`pip install erdscope`, then `erdscope demo`). Optional MySQL/PostgreSQL drivers, YAML parsing, Excel round-trip tests, and browser E2E each have separate dependencies; the core CLI, SQLite, HTML, and Excel writer use the standard library.

## Product model

Three provider kinds contribute one schema IR:

```text
database (physical catalog)
  → application model overlays (logical semantics)
  → config tables/relations (explicit final corrections)
  → merged IR
  → interactive HTML + optional Excel
```

This visual order is useful but simplified: field authority is domain-specific. Config has final authority; databases outrank frameworks for physical fields; frameworks outrank databases for logical comments/names. Repeated model sources resolve equal-priority conflicts in command order. See [Schema and merge model](domain/schema-merge.md).

At least one database, model path, or non-empty config `tables:` map is required. The DB is optional, so code-only and config-only generation do not connect or prompt for credentials.

## Repository rules that prevent common mistakes

1. **Edit `src/erdscope/*`, not `erd.py`.** `erd.py` is a deterministic committed build artifact generated by `tools/build_single_file.py`.
2. **The split files form one flat module.** Assembly order in `tools/build_single_file.py::MODULES` is import/runtime order; this is not a conventional package.
3. **Rebuild after Python or viewer changes:** `python3 tools/build_single_file.py`.
4. **Regenerate the hosted demo after viewer/output changes:** `python3 docs/gen_demo.py`; CI diffs `docs/index.html`.
5. **Keep optional dependencies optional.** CI first tests with none installed.
6. **Treat config changes as a merge-contract change.** Syntax validation, semantic validation, authority, operation stripping, and output serialization must stay aligned.
7. **Load only trusted plugins.** `--adapter` executes Python in-process and later registrations may override built-ins.

## Change routing

| Goal | Start with | Minimum focused verification |
|---|---|---|
| Change CLI/provider flow | `src/erdscope/cli.py`, `providers.py` | `tests.test_pipeline` |
| Change merge/config semantics | `merge.py`, `config.py`, `providers.py` | merge + config + pipeline + characterization suites |
| Add/change DB engine | `db/base.py`, engine fragment | adapter units; real integration for server DBs |
| Add/change model parser | `frameworks/base.py`, overlay fragment | parser fixtures + pipeline registry tests |
| Change viewer/layout/export UI | `viewer.html` | Playwright E2E, rebuild, hosted demo regeneration |
| Change Excel workbook | `exporters.py` | exporter tests, optional openpyxl round trip |
| Change demo | `demo.py`, example/docs generators | demo tests and regenerated assets |
| Release | `pyproject.toml`, `CHANGELOG.md`, release workflow | full tests, build check, demo diff, clean tree |

## Architecture at a glance

- `pyproject.toml` packages only `erd` and exposes `erd:main`.
- `tools/build_single_file.py` concatenates ordered Python fragments and inlines `viewer.html` into `exporters.py`’s sentinel.
- `cli.py::_run_pipeline()` loads config/plugins, collects provider layers, merges and validates.
- `merge.py::merge_ir()` is the central pure merge; it reconciles duplicate DB-FK edges and computes derived state.
- `cli.py::_finish()` applies optional inference/filtering, converts internal provenance to viewer flags, then writes HTML/Excel.
- Registries in `db/base.py` and `frameworks/base.py` provide built-in and runtime plugin seams.

Recent history explains the shape: the pipeline was refactored into layered IR to enable DB-less sources; source was split without abandoning single-file distribution; registries enabled pluggability; SQLite and a truly installed-safe demo followed; real DB CI and large-schema benchmarks then hardened parity and practical limits (`CHANGELOG.md` and targeted recent commits).

## Operational baseline

```bash
# full optional browser support
pip install playwright openpyxl pyyaml
playwright install chromium
python3 -m unittest discover -s tests -v

# deterministic artifact checks
python3 tools/build_single_file.py --check
python3 docs/gen_demo.py
git diff --exit-code docs/index.html
```

Real DB tests use `ERDSCOPE_IT_MYSQL_URL` and `ERDSCOPE_IT_POSTGRES_URL`. Credentials must not be stored in config; use read-only accounts and standard environment/client authentication. Performance is browser-bound: the documented conservative interactive ceiling is roughly 900 tables, after which generation-time `--only`/`--exclude` is recommended.
